# Tour of goes plan

## Goal

Build an interactive `Tour of goes` inside the existing VitePress docs, backed by a separate remote Go runner. The tour should be the hands-on path, while `Getting Started`, `Tutorial`, and `Guide` remain the deeper reference material.

## Product shape

- Add a top-level `Tour` section to the docs with a landing page and 4 runnable lessons.
- Each lesson page should contain: a short objective, 1-2 editable files, `Run` and `Check` actions, output/check panels, reset action, and links to the matching docs chapter.
- Keep the lessons narrow and self-contained; no account system, saved progress, sharing, or freeform project creation in v1.

## Lesson set for v1

1. `quick-start`
   - Based on `docs/getting-started/quick-start.md`
   - Todo aggregate plus in-memory save/fetch

2. `first-aggregate`
   - Based on `docs/tutorial/02-first-aggregate.md`
   - `aggregate.Base`, DTOs, `event.ApplyWith`, `aggregate.Next`

3. `events-and-state`
   - Based on `docs/tutorial/03-events-and-state.md`
   - Raise multiple events and apply state transitions

4. `repository`
   - Based on `docs/tutorial/05-repository.md`
   - Save, fetch, replay, and typed repositories

- Defer commands, projections, testing, and production backends until the runner and lesson authoring flow are stable.

## Architecture

### Docs UI

- Keep the docs site in `docs/`; extend the VitePress theme instead of building a separate frontend app.
- Modify:
  - `docs/package.json`
  - `docs/.vitepress/config.ts`
  - `docs/.vitepress/theme/index.ts`
  - `docs/.vitepress/theme/style.css`
  - `docs/.vitepress/theme/HomePage.vue`
  - `docs/index.md`
- Add interactive tour UI under `docs/.vitepress/theme/components/tour/` and shared client logic under `docs/.vitepress/theme/composables/`.
- Add `docs/tour/index.md` plus one markdown page per lesson; these pages should stay thin and mostly mount reusable tour components.
- Add a runner base URL config via docs environment variables so the static docs site can talk to the separately deployed runner service.

### Lesson source of truth

- Add a top-level `tour/lessons/<lesson-id>/` directory for each lesson.
- Each lesson should contain:
  - a manifest with title, goal, docs links, editable files, run/check commands, and file ordering
  - a starter workspace Go module
  - locked harness files such as `cmd/app/main.go` or hidden check files
- Use multi-file lessons, but expose only 1-2 editable files in the UI.
- Send the full editable file contents on each request so lessons are stateless and resettable.

### Runner service

- Add a small HTTP service at `cmd/tour-runner/main.go` with implementation packages under `internal/tourrunner/`.
- The service should accept `lessonId`, edited file contents, and an action (`run` or `check`).
- On each request it should:
  - load the lesson template
  - materialize a temporary workspace
  - write edits only into allowed files
  - execute the lesson command in an isolated container
  - return structured JSON results
- Add runner container assets under `tour/runner/`.

### Result model

- Return structured data the docs UI can render directly:
  - `status`
  - `stdout`
  - `stderr`
  - `durationMs`
  - `compileErrors`
  - `checks[]` with `id`, `label`, `status`, and `message`
- `Run` should show visible execution output.
- `Check` should run hidden validation, preferably tests rather than brittle stdout matching.

## Isolation and safety requirements

- Do not run user code directly on the host.
- Use a short-lived container per request with:
  - no outbound network
  - strict CPU, memory, pid, and timeout limits
  - read-only base image
  - writable temp workspace only
  - no user-chosen commands or dependency installs
- Pin the Go toolchain and lesson environment so results stay reproducible.

## Implementation order

1. Define the lesson manifest schema and `tour/lessons/` layout.
2. Build the runner service and container executor around a single lesson.
3. Add the VitePress lesson shell, runner client, and landing page.
4. Author the 4 initial lessons and wire homepage/nav discovery.
5. Add automated tests, fixture validation, and end-to-end smoke coverage.

## Verification

1. Runner unit tests
   - Add Go tests for request validation, lesson loading, workspace assembly, timeout handling, result parsing, and container execution boundaries.

2. Lesson fixture tests
   - Verify every lesson manifest references real files, only exposes allowed editable paths, and has a starter state that fails/passes as intended.

3. Local end-to-end flow
   - Start the runner locally.
   - Run the docs site from `docs/`.
   - Open each `/tour/` lesson, edit code, run it, check it, reset it, and verify links back to the existing docs.

4. Build verification
   - Run `npm run build` in `docs/`.
   - Run Go tests for the new runner packages.
   - Build the runner container image and issue at least one sample request against a lesson.

5. Browser/manual checks
   - Confirm desktop and mobile layouts.
   - Confirm loading, error, timeout, and failed-check states in the UI.
   - Confirm the homepage and nav clearly position `Tour` as the interactive path and the existing docs as deeper reading.

## Scope cuts to protect v1

- No persisted user progress
- No auth or multi-user features
- No arbitrary code execution outside lesson templates
- No external package installs or outbound network access
- No advanced lessons until the first 4 are stable
- No duplication of long-form docs; the tour should link back to existing chapters for explanation

## Main risk

The highest-risk part is secure remote code execution. The runner must be sandboxed first; if that cannot be made safe and reliable, the tour should not ship.
